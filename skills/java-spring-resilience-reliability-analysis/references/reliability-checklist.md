# Spring Microservice Reliability Checklist

## 1. 请求入口与边界保护

- 是否为所有同步调用设置了明确超时（连接超时、读超时、总超时）
- 是否设置全局与接口级限流（含突发流量策略）
- 是否对高风险接口设置并发隔离（线程池/信号量）
- 是否定义拒绝策略后的业务降级响应

## 2. 下游调用治理

- 是否启用熔断器并配置开启阈值、半开探测与恢复窗口
- 是否启用重试并配置指数退避+随机抖动
- 是否限制重试的异常类型与状态码，避免无效重试
- 是否区分读写操作的重试策略（写请求优先幂等化后再重试）

## 3. 数据一致性与幂等

- 是否为跨服务写操作提供幂等键（业务主键、请求号、去重表）
- 是否定义最终一致性方案（Outbox、Saga、补偿事务）
- 是否存在消息重复消费保护与死信重试上限
- 是否提供手工修复或自动对账流程

## 4. 资源池与背压

- 线程池核心参数是否基于压测校准（核心线程、队列长度、最大线程）
- 连接池是否配置最大连接、等待超时、泄露检测
- 是否设置批处理/消费速率上限，避免压垮下游
- 是否存在队列积压监控与自动扩缩容策略

## 5. 可观测与告警

- 是否有四大黄金信号：延迟、流量、错误、饱和度
- 是否按接口/依赖维度暴露成功率与尾延迟（P95/P99）
- 日志是否包含可追踪关联 ID（traceId、spanId、requestId）
- 告警是否具备抑制与聚合，避免告警风暴

## 6. 发布与恢复

- 是否有灰度发布与按比例放量机制
- 是否定义自动回滚条件（错误率、延迟、关键业务指标）
- 是否定期执行故障演练（依赖超时、网络分区、消息堆积）
- 是否有可执行 Runbook（处理步骤、负责人、升级路径）

## 7. 评分建议

- P0：会导致核心链路不可用或数据破坏，需立即治理
- P1：高概率放大故障或导致明显 SLA 违约，短期治理
- P2：非阻断风险，纳入迭代治理
