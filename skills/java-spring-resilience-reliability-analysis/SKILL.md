# Java Spring Resilience Reliability Analysis

## 目标

对 Spring Boot/Spring Cloud 微服务执行结构化可靠性审查，优先发现会导致级联故障、数据不一致、雪崩、长尾延迟放大和不可恢复故障的设计缺陷。

## 执行流程

1. 识别系统上下文：
   - 服务边界、关键链路、依赖拓扑（DB/MQ/缓存/第三方 API）。
   - 可用性目标（SLA/SLO）和业务恢复目标（RTO/RPO）。
2. 盘点故障控制点：
   - 入站与出站超时、重试、熔断、限流、舱壁隔离、降级、回退。
   - 线程池/连接池/队列容量与拒绝策略。
3. 审查一致性与恢复：
   - 幂等键、去重窗口、事务边界、补偿流程、死信处理。
   - 启动恢复、重放机制、状态修复脚本与演练证据。
4. 审查可观测与运营：
   - 关键指标、日志关联 ID、分布式追踪、告警噪声控制。
   - 灰度发布、回滚路径、故障演练与容量压测结论。
5. 产出分级结论：
   - 每条问题必须包含影响范围、触发条件、当前证据、修复建议。
   - 建议按 P0/P1/P2 排序，先处理会导致全链路不可用的问题。

## 重点检查项

按需加载 `references/reliability-checklist.md`，逐项对照并记录“已满足/部分满足/缺失”。

## 判定规则

1. 任何关键链路无超时上限，判定至少 P1。
2. 重试无幂等保障或无退避策略，判定至少 P1。
3. 单实例依赖（缓存、锁、配置中心、消息中间件）无降级或故障转移，判定至少 P1。
4. 熔断/限流缺失且下游不可控（第三方 API、跨域网络调用），判定至少 P1。
5. 无可执行回滚方案或未演练，判定至少 P1。

## 输出格式

按以下结构输出，避免泛化建议：

1. `System Snapshot`：架构与关键链路摘要。
2. `Findings`：按 P0/P1/P2 列出问题，包含证据与触发场景。
3. `Fix Plan`：每条问题给出实现级修复方案（配置、代码、治理动作）。
4. `Validation Plan`：如何通过压测、故障注入、回归测试验证修复生效。
5. `Rollout Advice`：灰度顺序、观测指标、回滚阈值。

## 约束

1. 不凭空假设系统能力；缺失信息明确标注为“待确认”。
2. 优先给出 Spring 生态可落地方案（Spring Retry、Resilience4j、Spring Cloud CircuitBreaker、Actuator/Micrometer）。
3. 建议必须可执行，包含关键参数方向（例如超时、重试次数、退避策略、线程池容量）。
