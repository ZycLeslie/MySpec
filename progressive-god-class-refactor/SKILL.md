# Progressive God Class Refactor

## 输入与边界

1. 先识别目标类文件路径与主语言（Java/Kotlin/C#）。
2. 再统计目标类有效代码行数（排除空行和纯注释行）。
3. 明确继承层次、接口实现和 override 方法集合。
4. 明确保留约束：
   - 不能改 public API 的签名与语义（除非用户明确允许）。
   - 不能改变 override 方法的声明、可见性、返回类型和调用契约。
   - 不能引入与现有框架冲突的生命周期副作用。

## 行数分级策略

### A. 500-1000 行：一次性直接重构

1. 直接在一次变更中完成结构拆分。
2. 优先按职责切分：
   - 领域计算逻辑
   - 数据访问/网关逻辑
   - 编排/流程控制逻辑
   - 格式化/转换工具逻辑
3. 允许同时进行：
   - 提取私有方法
   - 提取同包新类（或内部类后再外提）
   - 消除重复分支
4. 保留入口方法行为不变，必要时在入口保留薄封装转发。

### B. 1000-2000 行：两次重构

按两次提交（或两批变更）完成，避免一次性风险聚合。

#### 第一次重构（结构铺垫）

1. 只做“低风险、等价”动作：
   - 提取纯函数/无副作用私有方法
   - 提取常量、配置映射、简单 DTO/VO
   - 将超长方法拆成可命名的小步骤
2. 不在第一次重构中改变核心调用拓扑。
3. 每次提取后保持测试可运行。

#### 第二次重构（职责迁移）

1. 在已有铺垫上迁移职责到新类：
   - 协调器/Facade 保留在原类
   - 业务规则、策略、转换器迁移到独立组件
2. 原类保留最小桥接代码，逐步减少字段与方法密度。
3. 在第二次结束后，确认依赖方向清晰（上层依赖抽象，下层依赖实现）。

### C. 超过 2000 行：仅拆非 override 方法

1. 禁止拆分或改写 override 方法体（除非只做机械性委托且行为严格等价）。
2. 仅对非 override 方法执行提取与迁移：
   - 将可独立职责迁移到 helper/service/strategy 类
   - 原类中保留同名薄封装，内部委托到新组件
3. 若非 override 方法内部依赖 override 行为：
   - 先抽取依赖接口（只读或回调）
   - 再迁移非 override 逻辑到新类
4. 避免触碰框架回调链、反射入口、注解驱动入口的签名与语义。

## override 识别规则

1. 语言显式标记（`@Override`、`override`）优先。
2. 无显式标记时，按父类/接口同签名方法推断 override。
3. 对“框架隐式 override”保持保守：
   - 生命周期方法
   - 事件回调方法
   - 序列化/反序列化钩子
4. 对疑似 override 方法默认“不迁移”，只允许内部提取局部私有函数。

## 执行流程

1. 先做静态盘点：
   - 方法清单（override / 非override）
   - 字段读写矩阵
   - 外部可见 API 清单
2. 按行数策略生成重构计划并声明当前阶段目标。
3. 逐步改造并在每阶段后运行最小验证集：
   - 单元测试
   - 编译检查
   - 关键回归路径（若有）
4. 输出结果时必须包含：
   - 本次改动范围
   - 未改动的高风险区
   - 下一阶段建议动作（若仍需继续）

## 代码变更准则

1. 优先“小步可回滚”，避免跨模块大爆炸式改动。
2. 新增类命名体现职责，不使用 `Util` 泛化命名（除非确实通用工具）。
3. 单个新增方法控制在可读范围，优先早返回减少嵌套。
4. 保留日志与异常语义，不吞异常，不改变错误码契约。
5. 若测试缺失，先补关键行为保护测试再迁移核心逻辑。

## 输出格式要求

执行此 skill 时，按以下结构回复：

1. `Class Size`：有效行数与分级结论（A/B/C）。
2. `Refactor Scope`：本轮将改动的职责与不改动区域。
3. `Changes`：新增/修改类与关键方法迁移说明。
4. `Validation`：已运行的验证与结果。
5. `Next Step`：若属于 B 或 C，给出下一步最小可执行计划。
