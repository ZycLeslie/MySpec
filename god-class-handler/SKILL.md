# God Class Handler

将本技能用于“超大类/超大方法”的可控重构，优先保证行为不变和调用迁移完整。

## 输入要求

开始前收集以下信息：

1. 目标文件路径与语言（Java/Kotlin/TypeScript/C#/Python 等）。
2. 目标类名、相关接口/父类、关键依赖。
3. 构建命令（至少一个可执行校验命令）与测试命令（可选）。
4. 允许的重构边界（是否允许改公共 API、是否允许新增文件）。
5. 测试执行策略（默认不执行；仅在用户明确确认后执行）。

若输入不完整，先做静态扫描并给出缺失项，再继续最小可行重构。

## 规模分流规则

将“类总行数 + 核心方法行数”一起评估，采用更严格的路径：

1. `<=500` 行  
   重点做“超大方法缩小 + 功能拆分”，通常由主 agent 直接完成。
2. `500-1000` 行  
   先划分功能模块，每个模块都委托子 agent 串行处理，主 agent 每轮验收后再进入下一模块。
3. `1000+` 行  
   使用主 agent + 子 agent 编排。所有责任模块均由子 agent 逐个处理；主 agent 在每轮结束执行遗漏检查和方法签名变更检查。
4. `2000+` 行  
   进入“强约束迁移模式”：禁止大批量一次性修改；按小批次方法簇串行迁移，每批必须提供迁移成功证据，主 agent 验证通过后才能进入下一批。

## 执行流程

### 步骤 1：盘点与切分（主 agent）

1. 枚举目标类中的：
   - 字段/常量
   - 公共方法
   - 私有方法
   - 外部调用点
2. 生成“功能单元图”（每个单元保持单一职责）：
   - 参数解析/校验
   - 业务编排
   - 持久化访问
   - 外部 API/基础设施调用
   - 结果组装/格式化
3. 对每个单元标记：
   - 依赖字段/常量
   - 被哪些方法调用
   - 是否可独立迁移

### 步骤 2：按阈值选择重构策略

#### A. `<=500` 行

1. 拆分超大方法为语义化私有方法（提取 early-return、校验、分支处理）。
2. 将重复逻辑提取为 helper/utility（仅在同一语义域内）。
3. 保持外部方法签名不变，优先做内部整理。

#### B. `500-1000` 行

1. 先定义模块边界（示例：`ValidationModule`、`ExecutionModule`、`MapperModule`）。
2. 每个模块都必须委托给子 agent，按模块顺序串行执行，不并行混改。
3. 每个模块完成后由主 agent 执行“模块验收门禁”：
   - 检查调用点是否遗漏
   - 检查方法签名是否发生未授权变更
   - 检查编译通过
   - 若用户已确认执行测试，再检查最小测试通过
4. 仅当当前模块通过门禁，才允许委托下一个模块。
5. 原类保留薄编排层，直到调用点迁移稳定后再收缩。

#### C. `1000+` 行

1. 主 agent 先产出“责任模块队列”，每个子任务只含一个功能单元。
2. 必须串行委托：一次只运行一个子 agent 任务，禁止多模块并行落地。
3. 每个子 agent 输出：
   - 新增/修改文件
   - 迁移的方法列表
   - 调用点替换列表
   - 方法签名前后对照表
   - 风险与待主 agent 处理项
4. 每个子任务结束后，主 agent 必须先验收再继续：
   - 调用点遗漏检测（基线对比 + 全文检索）
   - 方法签名变更检测（public/protected/internal）
   - 构建校验
   - 若用户已确认执行测试，再做最小回归测试
5. 全部模块完成后，主 agent 做最终收尾：
   - 从原类删除已迁移方法
   - 删除原类中不再使用的常量/字段
   - 清理无用 import/依赖
   - 执行全量调用检查与回归测试

#### D. `2000+` 行（强约束迁移模式）

1. 将每个责任模块再切成“方法簇批次”（建议每批 3-8 个方法，且共享依赖）。
2. 每批只允许一个子 agent 任务，禁止跨批次修改。
3. 子 agent 必须输出“迁移成功证据包”：
   - 新类/新模块中的方法实现位置
   - 原类中旧实现的处理结果（已删除或仅保留薄委托）
   - 调用点替换结果
   - 实现完整性证明（非空实现、无占位语句）
4. 主 agent 对每批执行“真实迁移校验”（见下文）并记录通过/失败。
5. 任一批失败，先返工当前批次，禁止推进下一批。
6. 所有批次完成后，主 agent 统一清理原类残留方法和未使用常量。

### 步骤 3：模块级串行委托循环（主 agent 必须执行）

对模块队列中的每个责任模块，重复以下循环：

1. 生成该模块的签名基线：
   - 导出当前对外方法签名清单（名称、参数、返回类型、可见性）。
2. 委托子 agent 执行“单模块迁移”。
3. 子 agent 完成后，主 agent 立即执行验收：
   - 先将该模块已重构方法回写到原类（替换原入口为新模块委托或兼容桥接）
   - 调用点遗漏检查通过
   - 签名变更检查通过（若有变更，必须有授权或兼容层）
   - 编译通过
   - 若用户已确认执行测试，再验证最小测试通过
4. 验收失败则回退到该模块修复，不进入下一个模块。
5. 验收通过后，更新迁移矩阵并委托下一个模块。

### 步骤 4：真实迁移校验（2000+ 强制，其他场景建议）

对每个“声明已迁移”的方法，主 agent 必须全部通过以下检查：

1. 实现落位检查：
   - 新模块中存在同语义实现（非空壳方法）。
   - 禁止占位实现：`TODO`、`pass`、空方法体、仅抛未实现异常等。
2. 原类剥离检查：
   - 原类中不再保留旧业务实现；若保留入口，只允许薄委托/兼容桥接。
3. 调用闭环检查：
   - 调用点指向新实现或桥接入口，不得继续直连旧实现。
4. 一致性检查：
   - 方法签名变化与迁移矩阵一致，且符合授权范围。

若任一检查失败，该方法判定为“未迁移成功”，当前模块/批次不得验收通过。

## 子 agent 委托规范

对子 agent 的任务描述必须包含：

1. 单一功能单元范围（禁止跨单元扩展）。
2. 明确输入/输出契约（方法签名、异常语义、返回值语义）。
3. 必做清单：
   - 迁移后保持行为一致
   - 补齐该单元全部调用点
   - 输出方法签名前后对照
   - 输出迁移成功证据（新实现位置、原类旧实现处理结果、调用点替换结果）
   - 确保迁移方法为完整可执行实现，禁止空实现或占位实现
   - 若用户要求测试，更新或新增最小必要测试
4. 禁止事项：
   - 修改其他单元逻辑
   - 无依据地变更公共 API
   - 跳过编译/测试验证

可复用模板见 [references/subagent-task-template.md](references/subagent-task-template.md)。

## 主 agent 收尾清单（必须执行）

1. 删除原类中“已迁移方法”的旧实现（禁止保留失效重复逻辑）。
2. 确认原类中的对应入口已指向子 agent 产出的新实现（委托/桥接），禁止悬空入口。
3. 删除原类未再引用的常量、字段、私有工具方法。
4. 使用全文检索确认调用迁移无遗漏（方法名、类名、旧常量名）。
   - 对 2000+ 场景，逐方法核对“新实现存在 + 原实现已删除或委托”。
5. 执行构建；并在用户确认时执行测试。覆盖范围：
   - 编译通过
   - 若执行测试：受影响模块测试通过
   - 若执行测试：关键路径 smoke test 通过
6. 输出迁移矩阵：
   - 旧方法 -> 新归属
   - 旧调用点 -> 新调用点
   - 删除项（方法/常量/字段）
7. 输出签名审计结果：
   - 未变更签名列表
   - 经授权的签名变更列表（含兼容策略）
8. 输出原类回写清单：
   - 已替换为委托的原类方法入口
   - 已保留兼容桥接的方法入口

## 防漏调用检查（强制）

对每个已迁移方法执行以下检查：

1. 迁移前记录调用点基线（`rg "<methodName>\\("`）。
2. 迁移后再次检索，确认旧调用已清零或仅保留兼容桥接。
3. 若存在反射/字符串调用，增加额外搜索关键字与运行时验证。
4. 对外暴露 API 若改名，必须提供迁移层或一次性全量替换并测试。

## 签名变更检查（强制）

对每个模块迁移后执行：

1. 对比迁移前后的方法签名清单（名称、参数、返回值、可见性、异常约定）。
2. 若发现未授权签名变更，判定该模块验收失败并返工。
3. 若签名必须变更，要求：
   - 提供兼容桥接层或适配器
   - 同步完成所有调用点替换
   - 在输出中标注变更原因和影响范围

## 质量门禁

任一项失败则不得结束：

1. 编译失败。
2. 用户已确认执行测试且关键测试失败。
3. 发现遗漏调用点。
4. 原类仍保留已迁移且不再使用的重复实现。
5. 原类中仍存在未使用常量且可安全删除。
6. 出现未授权方法签名变更。
7. 子 agent 已重构的方法未被主 agent 回写到原类入口。
8. 任一声明已迁移方法未通过“真实迁移校验”。
9. 发现空实现、占位实现或未完成实现被当作迁移结果提交。

## 输出格式（每次执行后）

1. 重构策略：`<=500` / `500-1000` / `1000+`
2. 模块划分与子任务列表
3. 已完成迁移方法清单
4. 已删除方法/常量/字段清单
5. 调用点迁移校验结果
6. 方法签名变更校验结果
7. 原类回写校验结果
8. 真实迁移校验结果（逐方法：成功/失败）
9. 实现完整性校验结果（是否存在空实现/占位实现）
10. 构建结果与测试执行状态（未执行/已执行）
11. 剩余风险与下一步
